## Шаги

Порядок работ для разработчиков: сначала стабилизация текущего прототипа, затем переход к целевой архитектуре из `WorkTest.md`, затем игровые и продуктовые фичи. Как поднять проект локально — в разделе «Как запустить текущий прототип».

### Стабилизация текущего стека (Express + статика)

1. Свести backend к одной точке входа; удалить или вынести из репозитория экспериментальные файлы (`qwerty.js`, дубликаты `index` и т.п.).
2. Настроить доступ к OpenAI только через переменные окружения; исправить подстановку ключа в коде (без строковых литералов с «псевдо-${...}»).
3. Убрать с клиента любые прямые вызовы API с секретами; весь ИИ — только через backend.
4. Объединить вызовы к модели в одну поддерживаемую интеграцию на актуальном API (чаты/комплишены — по выбранной схеме).
5. Добавить в `package.json` осмысленные скрипты (`start`, при необходимости `dev`) и при желании минимальные проверки качества (линтер, форматтер).

### Переход к целевой архитектуре (`WorkTest.md`)

6. Заложить структуру репозитория: `frontend/` (Next.js + TypeScript, Tailwind, shadcn/ui), `backend/` (FastAPI), `infra/` (compose для Postgres, Redis, Ollama, ComfyUI и др. по `spec/`).
7. Описать и реализовать `docker-compose` для инфраструктуры; зафиксировать в `WorkTest.md` или `spec/` ответственность каждого сервиса.
8. Перенести или переписать серверную логику на FastAPI (uvicorn, единый слой для LLM и будущих эндпоинтов).
9. Перенести UI на Next.js: маршруты, компоненты, связь с API; выровнять порты и прокси под dev и будущий прод.
10. Настроить удобный цикл разработки: автоперезапуск backend, hot reload frontend, при необходимости общий Makefile или документация по порядку сервисов.

### Игровая логика и продукт

11. Состояние сессии и история сообщений на сервере (и контракт для клиента).
12. Базовый игровой цикл (ходы, реакции ведущего, привязка к контексту квеста).
13. Механики: броски костей, проверки навыков/характеристик, последствия.
14. Персистентность: персонаж, прогресс квеста, при необходимости память мира.
15. UX длительных сессий: навигация, сохранение, читаемость длинного текста.
16. Спроектировать и реализовать на backend стратегию **длинного контекста** для языковых моделей (подходы — в подразделе ниже); зафиксировать в `spec/` формат состояния сессии, лимиты токенов и то, что именно попадает в промпт.

### Длинный контекст для языковых моделей

Обычно используется **комбинация** приёмов, а не один трюк:

- **Бюджет и модель** — заранее задать лимит токенов на запрос (под окно выбранной модели), считать или оценивать размер истории; при необходимости перейти на модель с большим контекстом или на локальный LLM (например, Ollama в `infra/`) с явными лимитами.
- **Скользящее окно** — в каждый вызов передавать системную инструкцию, компактные блоки настроек (мир, персонаж) и только **последние N** реплик; сырую переписку целиком не копить бесконечно в промпте.
- **Скользящие саммари** — периодически сжимать ушедшие в прошлое ходы в краткую хронику (отдельным вызовом к модели или эвристикой), хранить её в состоянии сессии и подмешивать вместе с недавними сообщениями.
- **Структурированная память** — ключевые факты (квест, NPC, локации, договорённости) держать в компактном виде (схема/JSON в БД), а в промпт отдавать **выжимку**, а не полный лог чата.
- **RAG по лору** — если объём текстов велик: индексировать фрагменты (векторный или полнотекстовый поиск), в контекст модели подставлять только **релевантные** выдержки по текущей сцене.
- **Стриминг ответа** — не увеличивает входной контекст, но снижает трение при длинных ответах; при необходимости сочетать с поэтапной генерацией.

## Цель проекта

Создать лёгкий соло-ролевой опыт, в котором:

- игрок настраивает своё приключение,
- ИИ описывает сцены и реагирует на действия,
- сессия ближе к настольному RPG, чем к обычному чат-боту.

В перспективе — соло-песочница в духе DnD на ИИ с:

- непрерывностью повествования,
- состоянием персонажа,
- памятью мира,
- прогрессом квеста,
- механиками: броски костей, проверки, последствия.

## Текущее состояние

Сейчас в проекте есть:

- простой фронтенд в браузере (статические страницы),
- backend на Node.js + Express,
- начальный слой для генерации ответов ИИ,
- многостраничный сценарий ввода данных приключения и экран квеста.

Эту версию стоит считать **рабочим черновиком концепции**, а не готовой игрой.

---

## Что уже сделано

### Фронтенд

Статические HTML/CSS/JavaScript:

- **Стартовая страница** — приветствие и вход в сценарий приключения.
- **Страница настройки приключения** — поля: мир/сеттинг, основной квест, имя, раса, класс, предыстория; проверка обязательных полей перед переходом.
- **Страница квеста** — область вывода рассказчика, ввод игрока, кнопка отправки, базовый блок «лист персонажа».

### Backend

- сервер на **Express.js**,
- **CORS** и разбор **JSON**,
- раздача статики фронтенда,
- черновой endpoint для отправки ввода игрока и получения ответа ИИ.

## Текущий пользовательский сценарий

1. Открыть стартовую страницу.
2. Перейти в мир / к настройке приключения.
3. Заполнить форму приключения и персонажа.
4. Перейти на экран квеста.
5. Ввести действие или реплику игрока.
6. Отправить сообщение на backend.
7. Получить ответ в стиле рассказчика от слоя ИИ.

## Чего не хватает / что не завершено

Прототипу ещё далеко до полноценного соло-DnD:

### Игровые системы

- нет настоящего движка прав DnD,
- нет системы бросков костей,
- нет проверок характеристик, спасбросков, боя, инвентаря,
- нет отслеживания состояния квеста,
- нет структурированной походовой системы.

### Повествование

- нет устойчивой памяти мира,
- нет долговременного сохранения диалога/сессии,
- нет выстроенной архитектуры промптов для стабильного поведения ведущего,
- нет чёткого разделения рассказа, механик и намерений игрока.

### Техническая зрелость

- нет тестов,
- нет настройки под продакшен-деплой,
- в `package.json` нет удобного набора скриптов,
- остаются экспериментальные и дублирующие файлы на backend,
- часть интеграции с OpenAI устарела и требует приведения в порядок,
- ключи и секреты должны задаваться **только** через переменные окружения.

### Известные проблемы

- отдельные файлы backend выглядят как черновики или наследие (`qwerty.js` и т.п.),
- логику OpenAI нужно собрать в одну аккуратную реализацию,
- часть кода опирается на старые completion-эксперименты,
- секреты и API-ключи не должны попадать в исходники и в клиентский JS,
- границы ответственности клиента и сервера пока размыты,
- каталог `client/src/` задуман под будущую структуру, сейчас не используется.

## Заметки по разработке

Репозиторий сейчас стоит воспринимать как:

- эксперимент уровня MVP,
- проверку сценария взаимодействия,
- основу для будущей ИИ-соло-RPG системы.

Уже видно главную идею: игрок задаёт настройки приключения и общается с ИИ-рассказчиком через простое веб-приложение.

Пока это **не** демонстрация полноценного, устойчивого соло-DnD на системных правилах.

## Видение

Конечная цель — не просто «чат в фэнтези-стиле», а:

- инструмент для соло-кампаний,
- рассказчик с памятью и последовательностью,
- ролевой опыт в браузере,
- сочетание истории и механик.
