# Scripted Realms – Software Design Document (SDD)

> Проект: Scripted Realms  
> Версия документа: 0.1 (prototype)  
> Статус: Draft / Prototype  

Этот документ описывает архитектуру и дизайн прототипа Scripted Realms в формате SDD.  
Фокус текущей версии – **один мир, один персонаж, одна сессия**, при этом архитектура сразу готова к расширению.

---

## 1. Цели и рамки (Scope)

### 1.1. Цель продукта

Scripted Realms – браузерная соло TTRPG‑система, в которой:

- игрок создаёт персонажа и задаёт основные параметры мира и квеста;
- ИИ‑ведущий (LLM) описывает сцены, последствия действий и реакции мира;
- система генерирует **изображение сцены после каждого действия**;
- базовая механика (статы/проверки/урон) реализована как детерминированный rules‑engine.

Проект должен:

- запускаться **локально и бесплатно** (Ollama, ComfyUI, Postgres, Redis);
- выглядеть как **инженерно зрелый прототип**, пригодный для превращения в продукт;
- не зависеть жёстко от конкретного LLM/изображений (адаптеры).

### 1.2. Область MVP

В рамках этого SDD MVP включает:

- один пользователь (локальный режим, без регистрации);
- один мир по умолчанию;
- одного персонажа;
- одну активную сессию;
- минимальный набор механик:
  - базовые характеристики персонажа (например, STR/DEX/CON/INT/WIS/CHA + HP),
  - простые проверки (ability checks, saving throws в усечённом виде),
  - простой боевой урон (модель «попал/промахнулся + урон»).
- поток взаимодействия:
  - создание персонажа и старт сессии,
  - игровой экран с логом, карточкой персонажа и картинкой,
  - ввод действия → ход системы → ответ ИИ + новая картинка.
- **стриминг текста нратора** (LLM) на фронт:
  - минимум – SSE (Server‑Sent Events) сверху HTTP,
  - возможность замены/расширения на WebSocket в будущем.

### 1.3. Вне рамок MVP (но запланировано)

- Мультипользовательский режим и учётки.
- Несколько миров и персонажей на пользователя.
- Полный охват SRD‑механик (уровни, архетипы, множество классов, сложный бой).
- Богатая система инвентаря, состояний, эффектов и журналов мира.
- Облачный деплой и горизонтальное масштабирование.

## 2. Правовой контекст (SRD и лицензии)

### 2.1. SRD и использование DnD‑механик

Механики и элементы правил берутся из **System Reference Document (SRD) 5.1/5.2**, опубликованных Wizards of the Coast под лицензией **Creative Commons Attribution 4.0 International (CC BY 4.0)**.

Проект:

- может **использовать и модифицировать** контент SRD, в т.ч. коммерчески;

- **не является** официальным продуктом и не аффилирован с Wizards of the Coast.

Рекомендуемая формулировка для отдельного LEGAL‑раздела в основном проекте:

### 2.2. Ограничения

- Нельзя создавать впечатление официального продукта.

## 3. Архитектура на высоком уровне

### 3.1. Общий обзор

Система делится на следующие подсистемы:

- **Frontend** – Next.js + TypeScript, Tailwind + shadcn/ui.
- **Backend API** – FastAPI (Python).
- **Rules Engine** – детерминированная логика боёвки и проверок.
- **Story Orchestrator** – сбор контекста, вызовы LLM, разбор ответов.
- **Memory Service** – Postgres + pgvector для памяти сессий и лора.
- **Image Service** – обёртка над ComfyUI для FLUX.1‑schnell.
- **LLM Adapter** – обёртка над Ollama (Qwen3.5 9B/4B).
- **Infra** – Postgres, Redis, Ollama, ComfyUI (docker‑compose).

Коммуникация:

- Frontend ↔ Backend API: HTTP (REST) + SSE (для стриминга текста).
- Backend API ↔ LLM Adapter: HTTP‑клиент к Ollama (stream API).
- Backend API ↔ Image Service: HTTP‑клиент к ComfyUI (workflow API).
- Backend API ↔ DB/Redis: синхронные запросы к Postgres/Redis.

### 3.2. Компонентная диаграмма (логическая)

- **Frontend (Next.js)**
  - UI слои:
    - Landing page (`/`)
    - Character & adventure setup (`/new`)
    - Session view (`/session/[sessionId]`)
  - Клиентские сервисы:
    - HTTP‑клиент для REST API
    - SSE‑клиент для стриминга текста

- **Backend API (FastAPI)**
  - HTTP‑эндпоинты:
    - создание персонажа/сессии,
    - приём действий,
    - стриминг ответов,
    - чтение истории.
  - Сервисные слои:
    - `SessionService` (управление сессиями),
    - `CharacterService` (персонажи),
    - `WorldService` (миры),
    - `EventService` (журнал событий).

- **Rules Engine**
  - Функции:
    - базовые ability checks / saving throws,
    - броски урона (d20/d6/etc.),
    - обновление HP/состояний.

- **Story Orchestrator**
  - Задачи:
    - собрать контекст (персонаж + мир + релевантные события),
    - преобразовать в промпт для LLM,
    - вызвать LLM (через LLM Adapter),
    - распарсить ответ:
      - основной текст сцены,
      - структурированные подсказки по механике (если есть),
      - подсказки для изображения (scene/image hints).

- **Memory Service**
  - Хранение:
    - `sessions`, `events`, `event_summaries`,
    - `lore_entries` с векторами (pgvector).
  - Retrieval:
    - подбор N релевантных записей к текущему ходу.

- **Image Service (ComfyUI)**
  - Пайплайны:
    - генерация master‑портрета персонажа,
    - img2img сцен на основе портрета/предыдущего кадра,
    - использование IP‑Adapter + ControlNet (позже).

- **LLM Adapter (Ollama / Qwen3.5)**
  - Endpoints:
    - текстовая генерация с **stream=true** (для стриминга),
    - возможно – отдельный light‑endpoint для служебных задач (summary и т.п.).

## 4. Модель данных (MVP)

> Примечание: ниже логическая модель, а не финальная SQL‑схема.

### 4.1. World

- `id` – UUID.
- `name` – строка (для будущей мульти‑мировости).
- `description` – текст (сеттинг).
- `created_at`, `updated_at`.

В MVP может существовать один мир по умолчанию с фиксированным `id`.

### 4.2. Character

- `id` – UUID.
- `world_id` – ссылка на `World`.
- `name`, `race`, `class`, `background`.
- `level` – целое, MVP: фиксированное значение (например, 1).
- `stats` – JSONB с полями вроде:
  - `str`, `dex`, `con`, `int`, `wis`, `cha`,
  - `max_hp`, `current_hp`.
- `portrait_image_id` – внешний ID/URL на master‑портрет.
- `created_at`, `updated_at`.

### 4.3. Session

- `id` – UUID.
- `world_id` – ссылка на `World`.
- `character_id` – ссылка на `Character`.
- `status` – enum: `active`, `finished`, `abandoned`.
- `created_at`, `updated_at`.

В MVP допустима только одна `active`‑сессия.

### 4.4. SessionEvent

- `id` – UUID.
- `session_id` – ссылка на `Session`.
- `index` – порядковый номер хода.
- `event_type` – enum:
  - `player_action`,
  - `narrator_response`,
  - `system`.
- `payload` – JSONB:
  - для `player_action`: текст действия, возможно тип (dialogue/explore/combat),
  - для `narrator_response`: финальный текст ответа, изменения статов, image hints.
- `created_at`.

### 4.5. EventSummary / LoreEntry (для pgvector)

- `id` – UUID.
- `world_id` / `session_id`.
- `text` – текст лора/суммаризованного события.
- `embedding` – вектор (`vector` тип через pgvector).
- `created_at`.

### 4.6. UploadedAdventure (кастомные истории) – FUTURE, частично в расширенном MVP

Пользователь может загрузить свою подготовленную историю/кампанию, которую LLM должен вести:

- **Описание** – один файл с текстом приключения (сюжет, сцены, подсказки):
  - поддерживаемые форматы: **PDF, Word (.docx), TXT, Markdown (.md)**.
- **Картинки** – архив (zip) с иллюстрациями, где **имена файлов** соответствуют действиям/сценам.

Логическая модель:

- `uploaded_adventures`
  - `id` – UUID.
  - `name` – название приключения (от пользователя).
  - `description` – краткое текстовое описание.
  - `story_file_path` – путь/URL к загруженному файлу истории (в файловой системе/облаке).
  - `story_file_type` – enum: `pdf`, `docx`, `txt`, `md`.
  - `images_archive_path` – путь/URL к исходному архиву картинок (для отладки/повторной обработки).
  - `status` – enum:
    - `uploaded` – загружено, в очереди на обработку,
    - `processed` – PDF/архив проиндексированы и готовы,
    - `failed` – ошибка обработки.
  - `created_at`, `updated_at`.

- `uploaded_adventure_assets`
  - `id` – UUID.
  - `adventure_id` – ссылка на `uploaded_adventures`.
  - `asset_type` – enum: `image`, `text_segment`.
  - `source_name` – исходное имя файла (например, `fight_bandits.png`).
  - `action_key` – ключевое действие/сцена, нормализованное имя для логики (например, `fight_bandits`).
  - `text` – опциональный фрагмент текста (для привязки к PDF‑сегменту).
  - `image_path` – путь/URL к распакованному изображению.
  - `embedding` – вектор для поиска по смыслу (опционально, позже).
  - `metadata` – JSONB (доп. инфа: страница PDF, позиция, теги).

### 4.7. Связь кастомных историй с сессиями – FUTURE

Для сессий, идущих по кастомным историям:

- в `Session` добавляется поле:
  - `uploaded_adventure_id` – опциональная ссылка на `UploadedAdventure`.
- возможен режим:
  - `session_mode` – enum: `generated_world` (по умолчанию) / `uploaded_adventure`.

## 5. API‑контракты (MVP)

Все пути предварительно, могут уточняться при реализации.

### 5.1. Создание персонажа и сессии

`POST /api/characters`

- Вход:
  - `name`, `race`, `class`, `background`,
  - базовые настройки (мир, описание квеста).
- Выход:
  - `character_id`.

`POST /api/sessions`

- Вход:
  - `character_id`,
  - (опционально) `world_id` – в MVP можно не передавать (берётся дефолт).
- Выход:
  - `session_id`.

### 5.2. Получение состояния сессии

`GET /api/sessions/{session_id}`

- Выход (упрощённо):
  - `session` – метаданные,
  - `character` – краткие данные (имя, HP, основные статы),
  - `last_events` – несколько последних событий (для быстрого восстановления UI),
  - `current_image_url` – ссылка на последнее изображение сцены (если есть).

### 5.3. История сессии

`GET /api/sessions/{session_id}/events?offset=&limit=`

- Пагинированная выдача истории событий, для длинных сессий.

### 5.4. Ход игрока (REST‑часть)

`POST /api/sessions/{session_id}/action`

- Вход:
  - `action_text` – строка (что делает герой),
  - (опционально) `action_type` – `"dialogue" | "explore" | "combat"` и т.п.
- Поведение:
  - создаётся `player_action`‑event,
  - инициируется ход Story Orchestrator и Rules Engine,
  - создаётся `narrator_response`‑event,
  - стартует/обновляется генерация изображения.
- Выход (быстрый отклик):
  - `event_id` для `player_action`,
  - `pending_response_id` или `response_event_id` (если успели завершить),
  - `image_request_id` (если картинка ещё рендерится).

### 5.5. Стриминг текста (SSE)

`GET /api/sessions/{session_id}/stream?response_id=...`

- Транспорт: **Server‑Sent Events (SSE)**.
- События:
  - `event: token` – очередной фрагмент текста:
    - `data: {"token": "часть текста", "response_id": "..."}`
  - `event: done` – завершение ответа:
    - `data: {"response_id": "...", "full_text_id": "event_id"}`.
  - `event: error` – ошибка генерации (по возможности с кодом/сообщением).

**Поток обработки:**

1. Клиент вызывает `POST /action` и получает `response_id` (или аналог).
2. Клиент открывает `EventSource` на `GET /stream?response_id=...`.
3. Backend:
   - вызывает LLM через LLM Adapter в режиме stream,
   - каждый токен пробрасывает как `event: token`,
   - по завершении шлёт `event: done`.
4. Клиент:
   - отображает текст по мере прихода токенов,
   - по `done` закрепляет полный ответ в логе.

### 5.6. Статус картинки

`GET /api/images/{image_request_id}`

- Выход:
  - `status`: `"pending" | "ready" | "failed"`,
  - `image_url` при `ready`.

Backend может также пушить обновления статуса другими способами (webhook/SSE), но для MVP достаточно периодического опроса.

### 5.7. Загрузка кастомных историй (Uploaded Adventures) – FUTURE

#### 5.7.1. Загрузка файла истории + архива картинок

`POST /api/uploaded-adventures`

- Тип запроса: `multipart/form-data`.
- Параметры:
  - `name` – строка (название истории).
  - `description` – строка (краткое описание).
  - `story_file` – файл истории (PDF/Word/TXT/Markdown).
  - `images_zip` – файл ZIP с картинками.
- Поведение:
  - проверяет тип файла истории (по расширению и/или MIME),
  - сохраняет файлы в файловой системе/хранилище,
  - создаёт запись в `uploaded_adventures` со статусом `uploaded`,
  - ставит задачу в очередь на обработку (парсинг текста из PDF/DOCX/TXT/MD, распаковка архива, индексация).
- Ответ:
  - `adventure_id`,
  - `status` (изначально `uploaded`).

#### 5.7.2. Статус обработки

`GET /api/uploaded-adventures/{adventure_id}`

- Ответ:
  - `adventure` – метаданные,
  - `status` – `uploaded` | `processed` | `failed`,
  - (опционально) краткая сводка по количеству картинок/сегментов.

#### 5.7.3. Использование приключения в сессии

- Расширение `POST /api/sessions`:
  - входной параметр `uploaded_adventure_id` (опциональный).
  - если указан:
    - создаётся сессия с `session_mode = uploaded_adventure`,
    - `world` может быть либо дефолтным, либо логически связанным с приключением.

#### 5.7.4. Правила именования картинок

PDF не содержит прямых ссылок на файлы; связь подсказывается названием файлов:

- пример имён:
  - `start_tavern.png` – стартовая сцена в таверне,
  - `fight_bandits.jpg` – бой с бандитами,
  - `after_battle_loot.png` – сцена после боя.
- при обработке:
  - имя файла нормализуется:
    - перевод в нижний регистр,
    - замена пробелов и спецсимволов на `_`,
    - обрезка расширения.
  - получается `action_key`, который используется:
    - как тег/ключ сцены,
    - как подсказка LLM, какую картинку использовать.

Если LLM в ответе укажет действие/сцену, совпадающую с `action_key`, Image Service может:

- либо напрямую взять соответствующую картинку вместо генерации,
- либо использовать её как **референс** в img2img‑пайплайне.

## 6. Внутренняя логика хода

### 6.1. Пайплайн обработки действия

1. **Приём действия**
   - Endpoint `POST /action` создаёт `player_action`‑event.
2. **Сбор контекста**
   - `SessionService`:
     - читает `Character`, `World`, последние N `SessionEvent`,
     - запрашивает из `MemoryService` релевантные `EventSummary` / `LoreEntry`.
3. **Вызов Rules Engine (до LLM, если нужно)**
   - для некоторых типов действий (например, явный “атакую”) может:
     - выполнить броски,
     - вычислить предварительные последствия (hit/miss, урон).
4. **Вызов LLM через Story Orchestrator**
   - собирается промпт:
     - краткое резюме мира,
     - краткое резюме персонажа,
     - выдержки из лора/предыдущих сцен,
     - текст последнего действия игрока,
     - (опционально) структурированная информация о результатах правил.
   - LLM вызывается **в stream‑режиме** (Ollama/Qwen3.5).
   - токены стримятся на фронт через SSE.
5. **Финализация ответа**
   - по завершении генерации:
     - текст сохраняется как `narrator_response`‑event,
     - при необходимости обновляются статы (если LLM подсказал механику),
     - создаётся/обновляется `EventSummary` (векторное представление) для памяти.
6. **Генерация изображения**
   - из ответа извлекаются `image hints` (через промпт‑инструкции к LLM).
   - Image Service запускает FLUX.1‑schnell пайплайн:
     - либо текст‑to‑image на основе master‑портрета,
     - либо img2img от предыдущей сцены.
   - по готовности `image_url` сохраняется в связи с `SessionEvent`.

## 7. Стриминг: технические детали

### 7.1. Backend → Ollama (LLM stream)

- LLM Adapter инкапсулирует:
  - построение тела запроса,
  - установку `stream=true`,
  - преобразование потока байт/JSON в последовательность токенов.
- Интерфейс адаптера (псевдо‑Python):

```python
async def stream_story(prompt: str, config: GenerationConfig) -> AsyncIterator[TokenChunk]:
    ...
```

где `TokenChunk` содержит:

- `text` – фрагмент строки,
- (опционально) `is_finished`.

### 7.2. Backend → Frontend (SSE)

- FastAPI‑endpoint использует streaming‑response:

- читает `TokenChunk` из LLM Adapter,
- для каждого токена пишет в HTTP‑ответ:

```text
event: token
data: {"token": "...", "response_id": "..."}

```

- по завершении:

```text
event: done
data: {"response_id": "...", "full_text_id": "..."}

```

### 7.3. Frontend (Next.js)

- На клиенте создаётся `EventSource`:

```ts
const es = new EventSource(`/api/sessions/${sessionId}/stream?response_id=${responseId}`);

es.addEventListener("token", (event) => {
  const data = JSON.parse(event.data);
  // обновить текст нарратора частично
});

es.addEventListener("done", (event) => {
  const data = JSON.parse(event.data);
  es.close();
  // зафиксировать финальный ответ
});
```

## 8. Frontend‑дизайн (MVP)

### 8.1. Страницы

- `/` – Landing:
  - краткое описание,
  - кнопка “Начать приключение”.

- `/new` – Character & Adventure Setup:
  - форма с полями:
    - `name`, `race`, `class`, `background`,
    - краткое описание мира (free‑text),
    - краткое описание квеста.
  - кнопка “Начать” → создаёт персонажа и сессию, делает redirect на `/session/[id]`.

- `/session/[sessionId]` – основной экран:
  - **левая часть**: лог текста (scrollable, последние N сообщений, автоскролл к низу).
  - **правая/верхняя часть**: карточка персонажа (HP, статы, краткое описание).
  - **правая/нижняя часть**: текущее изображение сцены.
  - внизу: поле ввода действия + кнопка отправки.

### 8.2. UX‑требования

- Чёткие статусы:
  - “Генерируем ответ нратора…”
  - “Рисуем сцену…”
- Не блокировать ввод следующего действия, пока картинка не дорисована (MVP‑решение).
- Возможность перескроллить лог и быстро прыгнуть к последнему сообщению.

## 9. Расширяемость и будущее развитие

### 9.1. Мульти‑миры и мульти‑персонажи

Архитектурные предпосылки:

- `world_id` в `Character` и `Session` уже заложен.
- миры и персонажи – отдельные сущности, а не “глобальные синглтоны”.

Для перехода к мульти‑мирам/персонажам:

- добавить UI‑экраны:
  - список миров,
  - список персонажей,
  - выбор мира/персонажа при старте сессии.
- в API:
  - фильтрацию по пользователю (когда появятся аккаунты),
  - расширенные параметры создания.

### 9.2. Аккаунты и авторизация

Планируемая модель:

- `User` + аутентификация (email/password, позже OAuth).
- Привязка `World`, `Character`, `Session` к `user_id`.
- JWT или session‑cookies для фронта.

### 9.3. Усложнение правил (SRD‑механики)

- Расширение `Rules Engine`:
  - уровни и опыт,
  - классовые способности,
  - заклинания,
  - сложный бой, инициативы, зоны, эффекты.
- Возможно, отдельный слой “rules DSL” или таблиц конфигурации, импортируемых из SRD‑совместимого формата.

### 9.4. Улучшение генерации изображений

- Добавление:
  - IP‑Adapter для удержания портрета,
  - ControlNet для поз/композиции,
  - переключение “быстрый кадр vs кинематографичный кадр”.
- Возможность ручного перегенерации сцены пользователем.

### 9.5. Масштабирование

- Выделение сервисов:
  - отдельный `llm-service`,
  - отдельный `image-service`,
  - pool LLM‑воркеров (возможно, через vLLM вместо Ollama).
- Горизонтальное масштабирование API и фронта.

## 10. Статус и ограничения

- Документ описывает **прототипную архитектуру**; детали имплементации (словари, конкретные промпты, конечные схемы таблиц) могут меняться.
- Реализуемые в MVP возможности:
  - один мир/персонаж/сессия, но с учётом будущего расширения;
  - стриминг текста ИИ‑нратора через SSE;
  - базовые SRD‑совместимые механики;
  - изображения сцен после каждого действия с использованием FLUX.1‑schnell.
- В дальнейшем документ должен эволюционировать:
  - с фиксацией принятых решений,
  - добавлением диаграмм (deployment, sequence),
  - конкретизацией договоров между сервисами (LLM/Image/Rules).

## 11. План разработки по итерациям

Этот раздел описывает, **в каком порядке** реализуются части системы. Каждая итерация должна быть работоспособной и давать ощутимый результат.

### 11.1. Итерация 0 – Инфраструктура и скелет репозитория

- Создать структуру проекта:
  - `frontend/`, `backend/`, `infra/`, `spec/`.
- Настроить `docker-compose` в `infra/`:
  - Postgres,
  - Redis,
  - Ollama (с моделью Qwen3.5 9B/4B),
  - ComfyUI (с FLUX.1-schnell).
- Проверить:
  - что все сервисы поднимаются локально,
  - есть доступ к Postgres/Redis,
  - Ollama и ComfyUI отвечают на тестовые запросы.

### 11.2. Итерация 1 – Backend: базовый FastAPI + модели данных

- Поднять минимальное приложение FastAPI:
  - health‑чек (`GET /health`),
  - подключение к Postgres и Redis.
- Описать ORM‑модели:
  - `World`, `Character`, `Session`, `SessionEvent`.
- Реализовать базовые миграции (Alembic или аналог).
- Реализовать первые API:
  - `POST /api/characters`,
  - `POST /api/sessions`,
  - `GET /api/sessions/{session_id}` (без LLM и картинок).

### 11.3. Итерация 2 – Frontend: минимальный UI и связка с API

- Создать Next.js приложение в `frontend/`.
- Страницы:
  - `/` – Landing,
  - `/new` – форма создания персонажа (подключена к `POST /api/characters` и `POST /api/sessions`),
  - `/session/[sessionId]` – заглушка с отображением базовых данных сессии (через `GET /api/sessions/{id}`).
- Настроить базовый UI‑фреймворк:
  - Tailwind CSS,
  - shadcn/ui.

### 11.4. Итерация 3 – LLM Adapter и стриминг текста

- Реализовать LLM Adapter к Ollama:
  - текстовая генерация,
  - поддержка `stream=true`.
- Добавить в backend:
  - endpoint `POST /api/sessions/{session_id}/action`,
  - endpoint `GET /api/sessions/{session_id}/stream?response_id=...` (SSE).
- На фронте:
  - в `/session/[sessionId]` реализовать:
    - поле ввода действия,
    - отправку на `POST /action`,
    - получение текста через SSE и отображение в логе.
- Проверить полный путь:
  - действие игрока → LLM → потоковый ответ в браузере.

### 11.5. Итерация 4 – Базовый Rules Engine и память сессии

- Реализовать минимальный Rules Engine:
  - базовые статы и HP,
  - простые проверки/урон.
- Расширить `SessionEvent`:
  - хранение изменений статов,
  - хранение итогового текста нратора.
- Добавить `EventSummary` / `LoreEntry` (логическая реализация):
  - простое сохранение текстов для будущего retrieval.
- На фронте:
  - отображение текущих статов персонажа и их изменений после хода.

### 11.6. Итерация 5 – Image Service и интеграция с ComfyUI

- Реализовать Image Service:
  - клиент к ComfyUI,
  - простой workflow для:
    - генерации портрета персонажа при создании,
    - генерации сцены после действия (text‑to‑image).
- Добавить API:
  - `GET /api/images/{image_request_id}` для статуса.
- На фронте:
  - в `/session/[sessionId]` показывать:
    - текущую картинку сцены,
    - состояние загрузки/ожидания.

### 11.7. Итерация 6 – Uploaded Adventures (кастомные истории)

- Backend:
  - реализовать модели `UploadedAdventure` и `uploaded_adventure_assets`,
  - endpoint `POST /api/uploaded-adventures` (приём PDF/DOCX/TXT/MD + zip с картинками),
  - endpoint `GET /api/uploaded-adventures/{adventure_id}` (статус).
- Реализовать фоновую обработку:
  - парсинг текста истории,
  - распаковка архива картинок,
  - нормализация имён файлов в `action_key`,
  - сохранение ассетов.
- Связь с сессией:
  - расширение `POST /api/sessions` параметром `uploaded_adventure_id`,
  - режим `session_mode = uploaded_adventure`.
- Frontend:
  - простой UI для загрузки приключения (минимально),
  - возможность стартовать сессию по загруженному приключению.

### 11.8. Итерация 7 – Улучшения UX и “полировка” прототипа

- Frontend:
  - улучшение лога (автоскролл, визуальное разделение ходов),
  - отображение статусов генерации текста/картинки,
  - базовые темы/стили для комфортной долгой сессии.
- Backend:
  - обработка ошибок и таймаутов LLM/Image сервисов,
  - логирование ключевых шагов (для дебага).

### 11.9. Дальнейшие итерации (за пределами первого прототипа)

- Мульти‑миры и мульти‑персонажи.
- Аккаунты и авторизация.
- Расширенная SRD‑механика (уровни, заклинания, сложный бой).
- Оптимизация image‑пайплайна (IP‑Adapter, ControlNet, “быстрый” vs “кино” кадр).
- Подготовка production‑деплоя.
